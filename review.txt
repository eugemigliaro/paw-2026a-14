THINGS TO FIX
=============

1. Recovery routes 403 for authenticated users — High
   webapp/.../config/SecurityConfig.java
   /verifications/**, /password-reset/**, /forgot-password and
   /register/resend-verification are gated with .anonymous(), so a logged-in
   user clicking a verification or password-reset link gets a 403 before the
   controller runs (violates AGENTS.md "preserve verification-link access").
   Fix: keep .anonymous() for /login and /register GET; move the email-driven
   recovery routes to .permitAll(). Add a test for the authenticated path.

2. register() has no exception catch — Medium
   webapp/.../controller/AuthController.java
   No try/catch around accountAuthService.register(...) and no @ControllerAdvice
   for EmailTakenException/UsernameTakenException/EmailPendingVerificationException.
   - TOCTOU race (two registrations of same email pass validation) → uncaught
     EmailTakenException → 500.
   - An unverified pending account now trips auth.registration.error.emailTaken
     instead of the "check your inbox / resend verification" affordance.
   Fix: handle the pending-verification case (restore resend affordance) and
   the duplicate exceptions gracefully.

3. Ownership check lives in the controller — Medium
   webapp/.../controller/UserModerationReportController.java:127-130
   Detail handler calls unscoped findReportById(reportId) then checks ownership
   in the controller. The scoped ModerationService.findReportByIdForReporter(
   reportId, reporter) exists but is never called.
   Fix: use the scoped service method; drop the controller-side ownership check.
   (Reported as being handled in a separate branch — confirm before duplicating.)

4. SearchFormValidator uses default-zone LocalDate.now() — Low
   webapp/.../validation/SearchFormValidator.java:35
   validateDateRange compares against raw LocalDate.now() instead of the
   PlatformTime zone (cf. CreateEventFormValidator), so a same-day UPCOMING
   search can be wrongly rejected on a UTC-deployed JVM.
   Fix: use the PlatformTime zone.

5. SecurityControllerUtils re-hardcodes role authorities — Low (cleanup)
   webapp/.../utils/SecurityControllerUtils.java
   authoritiesFor(...) re-hardcodes "ROLE_ADMIN_MOD"/"ROLE_USER", duplicating
   SecurityAuthorities.forRole(...).
   Fix: call SecurityAuthorities.forRole(...) and delete the private helper.

6. Unused latitude/longitude parameters in distance helpers — Low (cleanup)
   webapp/.../utils/EventCardAttributeUtils.java:44
   matchDistanceLabels/tournamentDistanceLabels take (latitude, longitude) but
   only read getDistanceKmFromViewer(); FeedController builds extra args to pass
   them. Fix: drop the unused params.

7. Same series-reservation query run twice per detail render — Low (efficiency)
   services/.../MatchServiceImpl.java:855
   buildSeriesReservationState and buildSeriesJoinRequestState each call
   ...ForSeries(seriesId, viewer, now) for the same render.
   Fix: reuse the already-shared occurrence set.

8. Overlapping permission/capability computation — Low (consistency)
   services/.../MatchServiceImpl.java
   getMatchManagementPermissions and the interaction-state path recompute the
   same host/manager/editable facts and disagree subtly (isMatchManagementLifecycleOpen
   vs isEditableMatch), so disabled-button state and the mutation guard can drift.
   Fix: derive one capability object once.
